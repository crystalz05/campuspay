1. Executive Summary
CampusPay is a mobile payment application built in Flutter targeting Nigerian polytechnic and university students. It centralizes three core financial activities students engage in daily: paying school fees via RRR reference numbers (Remita), purchasing mobile data bundles, and transferring funds between peers on campus.
The application is developed as a Final Year Project at Auchi Polytechnic, simulating a production-grade payment ecosystem in a sandbox/mock environment. The goal is to demonstrate end-to-end mobile payment flows, authentication, transaction management, and a clean, intuitive UX that reflects real-world fintech product standards.
2. Problem Statement
2.1 The Pain Points
Nigerian students currently deal with a fragmented payment experience:
School fees require visiting a bank or navigating the REMITA web portal — no mobile-native, student-friendly flow exists.
Data bundle purchases happen across multiple informal channels (USSD codes, third-party apps, vendors) with no transaction history.
Peer-to-peer transfers rely on general banking apps not designed for the student context.
There is no single platform that unifies these three use cases under one student identity.
2.2 The Opportunity
A lightweight, well-designed mobile app that simulates this unified experience represents a compelling final year project, and a realistic prototype that could be pitched to polytechnic management or a startup. Even in sandbox/mock mode, it demonstrates genuine product thinking and technical depth.
3. Goals & Non-Goals
3.1 Goals
Implement school fee payment using Remita Flutter widget with real RRR number validation (sandbox environment).
Implement data bundle purchase with a mocked provider API (simulated success/failure responses).
Implement wallet-to-wallet fund transfer between registered CampusPay users.
Build a complete auth flow: registration, login, email verification, password reset.
Maintain full transaction history per user with status tracking.
Deliver a polished, production-quality UI worthy of a portfolio and final year defence.
3.2 Non-Goals
Live payment processing with real money — all transactions run in sandbox/mock mode.
iOS support — Android only for this project scope.
Third-party bank account linking or card tokenization.
Airtime top-up (out of scope to keep delivery focused).
Admin dashboard or institutional back-office panel.

5. Feature Specifications
5.1 Authentication Module
Stack: Supabase Auth (email/password) + deep link handling for verification and password reset.
Registration: name, email, password, matric number, institution name.
Email verification flow with deep link redirect back to app.
Login with session persistence (Supabase session tokens).
Password reset via email deep link (sabistyle:// scheme will be adapted).
Auto-logout on token expiry with graceful re-auth prompt.
5.2 Wallet / Dashboard
The central screen of the app. Shows the student's current simulated wallet balance and quick-action shortcuts.
Wallet balance display (virtual balance, seeded on registration e.g. ₦10,000).
Quick action buttons: Pay Fees, Buy Data, Transfer, History.
Recent transactions widget (last 5 transactions).
User greeting with avatar initials (no image upload needed for MVP).
5.3 School Fee Payment (Remita)
Core feature. Uses the official Remita Flutter widget in sandbox mode against remitademo.net.
Input screen: student enters RRR (Remita Retrieval Reference) number.
RRR validation: confirm the reference resolves to a valid fee record before proceeding.
Amount confirmation screen: display fetched amount, purpose, and institution name.
Payment screen: launch Remita Flutter widget (sandbox) to process payment.
Success/failure screen with receipt summary.
Transaction saved to Supabase with status, amount, RRR, and timestamp.
5.4 Data Bundle Purchase (Mocked)
Simulates a data bundle provider API. No real API integration — responses are mocked locally or via a simple Supabase Edge Function that returns simulated results.
Network selection: MTN, Airtel, Glo, 9mobile.
Phone number input with basic format validation.
Bundle selection: curated list of packages (e.g. 1GB/₦300, 2GB/₦500, 5GB/₦1,000).
Wallet balance check before proceeding — reject if insufficient.
Mock API call with 1–2 second delay to simulate network request.
90% simulated success rate, 10% simulated failure (for realistic UX testing).
Deduct amount from virtual wallet on success; update transaction history.
5.5 Peer-to-Peer Transfer
Allows registered CampusPay users to transfer virtual funds to each other.
Recipient lookup: search by registered email or matric number.
Recipient confirmation screen: show name and institution before transfer.
Amount entry with wallet balance validation.
Optional transfer note/memo.
PIN confirmation (4-digit transaction PIN set during onboarding).
Instant wallet debit/credit with transaction record for both parties.
Push notification to recipient on successful transfer (local notification simulation).
5.6 Transaction History
Full log of all user transactions across all three payment types.
Filterable by type: All, Fees, Data, Transfers.
Each entry: date, type, amount, status (success/failed/pending), description.
Tap to expand: full receipt detail view.
Pull-to-refresh for latest data.
5.7 Profile & Settings
View and edit display name.
Change transaction PIN.
Change password (via email reset flow).
Logout.
App version and build info (useful for defence demo).
6. Core User Flows
6.1 Onboarding Flow
Splash Screen  →  Onboarding Slides (3)  →  Register / Login  →  Email Verification  →  Set Transaction PIN  →  Dashboard
6.2 School Fee Payment Flow

6.3 Data Bundle Flow
Dashboard  →  Buy Data  →  Select Network  →  Enter Phone  →  Select Bundle  →  Confirm  →  PIN Entry  →  Mock Processing  →  Success/Fail  →  Wallet Deducted

8.2 RLS Policies
users: Users can only SELECT and UPDATE their own row.
transactions: Users can only SELECT their own transactions; INSERT allowed for system-level operations via service role.
transfers: Both sender and receiver can SELECT the transfer record.
notifications: Users can only access their own notifications.

10. Non-Functional Requirements
10.1 Performance
App cold start under 3 seconds on a mid-range Android device (4GB RAM).
All screen transitions under 300ms.
Transaction list loads within 1.5 seconds for up to 100 records.
10.2 Security
Transaction PIN stored hashed — never in plaintext.
Supabase RLS enforced on all sensitive tables.
JWT token stored in flutter_secure_storage, not SharedPreferences.
All API calls over HTTPS; no sensitive data in URL query params.
10.3 Reliability
Graceful error states for all network failures — no unhandled exceptions visible to user.
Idempotent transaction creation — no duplicate charges if user retries.
Optimistic UI for wallet deduction — rollback on failure.
10.4 Usability
All core flows completable within 4 taps from the Dashboard.
Accessible colour contrast (WCAG AA minimum).
Explicit loading indicators on all async operations.
Empty states for transaction history and search results.

A few things worth noting:
process_transfer RPC is critical. Never do the wallet debit/credit from Flutter-side with two separate updates — race conditions will bite you. This function handles the entire transfer atomically with a row lock on the sender. Call it from Dart via supabase.rpc('process_transfer', params: {...}).
security definer on the RPC means it runs with elevated privileges, bypassing RLS for the internal wallet updates. That's intentional — the RLS policies on users would otherwise block the receiver credit since the caller isn't the receiver. The function itself enforces the business logic, so it's safe.
Wallet seeded at ₦10,000 on registration so your demo flows don't immediately hit zero balance. You can adjust that default.
no_self_transfer constraint on the transfers table saves you from having to validate that in Flutter — the database rejects it outright.
